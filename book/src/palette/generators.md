# Theme generators

> Generators translate one palette into many config files — kitty, fish, and
> kakoune each get the same colors in their native format.

## What generators do

A generator takes a validated palette and produces a complete configuration
fragment for one application. Garden currently ships three generators:

| Generator | Output file | Format |
|-----------|------------|--------|
| kitty | `kitty/garden-theme.conf` | Kitty color conf (`#rrggbb`) |
| fish | `fish/garden-theme.fish` | Fish universal vars (bare hex) |
| kakoune | `kak/garden.kak` | Kakoune face declarations (`rgb:rrggbb`) |

Run all generators for the active palette:

```bash
just themes generate
```

Generate for a specific palette:

```bash
just themes generate kinu
```

Generate *and* apply to all running applications:

```bash
just themes try yoru
```

Generated files land in `/tmp/garden-themes/` by default. The `try` recipe
generates and then applies — it sends colors to kitty via `kitty @ set-colors`,
sources the fish file to set universal variables, and pipes the kakoune
colorscheme into all running kakoune sessions.

## What the output looks like

### Kitty

The kitty generator maps the 13 roles to kitty's special colors and the 16 ANSI
slots:

```conf
# Garden theme — mokume ◐
# Generated by garden-themes. Do not edit by hand.

# Special colors
background #2c3444
foreground #d4c5a9
cursor #d4c5a9
cursor_text_color #2c3444
selection_background #3d4759
selection_foreground #d4c5a9
url_color #c9b88c

# Normal colors (0–7)
color0 #252d3b
color1 #c4796b
color2 #7c9a7c
color3 #c9b88c
color4 #6b7a8d
color5 #4a5568
color6 #8b9bb0
color7 #8b9bb0

# Bright colors (8–15)
color8 #4a5568
color9 #c4796b
color10 #7c9a7c
color11 #c9b88c
color12 #8b9bb0
color13 #6b7a8d
color14 #d4c5a9
color15 #d4c5a9
```

The ANSI mapping uses the text hierarchy for blue/cyan/white slots and repeats
semantic colors (urgent, ok, accent) at the same value for normal and bright,
since Garden doesn't distinguish "bright red" from "red" — both mean `urgent`.

### Fish

Fish uses bare hex (no `#` prefix) and `set -U` for universal variables that
persist across sessions:

```fish
# Garden theme — mokume ◐
# Generated by garden-themes. Do not edit by hand.

set -U fish_color_normal 8b9bb0
set -U fish_color_command d4c5a9
set -U fish_color_keyword 6b7a8d
set -U fish_color_quote c9b88c
set -U fish_color_error c4796b
set -U fish_color_param 8b9bb0
set -U fish_color_comment 505e70
set -U fish_color_autosuggestion 505e70
set -U fish_color_selection --background=3d4759
set -U fish_color_operator 6b7a8d
set -U fish_color_escape c9b88c
set -U fish_color_redirection 6b7a8d
set -U fish_color_end 6b7a8d
```

## Design rationale

### One source of truth

Without generators, you'd maintain color values in three separate config files.
Change one hex value in your palette and you need to update kitty.conf,
garden-theme.fish, and garden.kak by hand. With four palettes and three
generators, that's 12 files to keep in sync.

The generator pipeline means `palettes.toml` is the single source. Change a
color there, run `just themes generate`, and every output file is consistent.

### Why a trait rather than templates?

Template engines (Tera, Handlebars) add a dependency and a new language to
learn. Each generator is ~50 lines of Rust with a `format!()` call. The logic is
simple enough that a string template would be more complex than the code it
replaces — especially for kakoune, where the face declarations need different
formatting rules for different face categories (some have attributes, some have
background colors, some have neither).

A trait also lets generators do computation. The kitty generator maps 13 roles
to 23 color settings (7 special + 16 ANSI). A template would need helper
functions for that mapping; a trait method just does it inline.

### Color format differences

Each target has its own hex color convention:

| Application | Format | Example |
|------------|--------|---------|
| kitty | `#rrggbb` | `#c4796b` |
| fish | bare hex | `c4796b` |
| kakoune | `rgb:rrggbb` | `rgb:c4796b` |

The `HexColor` type provides both `as_str()` (`#rrggbb`) and `bare()` (`rrggbb`)
accessors. Generators pick the accessor that matches their format, and kakoune
prepends `rgb:` in its format string.

## Implementation

### The ThemeGenerator trait

```rust
pub trait ThemeGenerator {
    /// Human-readable name (e.g., "kitty").
    fn name(&self) -> &str;

    /// Relative output path (e.g., "kitty/garden-theme.conf").
    fn relative_path(&self) -> &str;

    /// Generate complete config file content from a palette.
    fn generate(&self, palette: &Palette) -> String;
}
```

Each generator is a unit struct that implements this trait. No state, no
configuration — just a pure function from palette to string.

### Writing a new generator

1. Create `crates/garden-themes/src/generators/yourapp.rs`
2. Define a unit struct and implement `ThemeGenerator`:

```rust
use garden_core::{ColorRole, Palette};
use super::ThemeGenerator;

pub struct YourApp;

impl ThemeGenerator for YourApp {
    fn name(&self) -> &str { "yourapp" }

    fn relative_path(&self) -> &str { "yourapp/theme.conf" }

    fn generate(&self, palette: &Palette) -> String {
        let c = |role: ColorRole| -> &str {
            palette.color(role).unwrap().as_str()
        };

        format!(
            "# Garden theme — {} {}\n\
             background={}\n\
             foreground={}\n",
            palette.name, palette.icon,
            c(ColorRole::Base),
            c(ColorRole::Text1),
        )
    }
}
```

3. Declare the module in `generators/mod.rs`:

```rust
pub mod yourapp;
```

4. Register it in `all()`:

```rust
pub fn all() -> Vec<Box<dyn ThemeGenerator>> {
    vec![
        Box::new(kitty::Kitty),
        Box::new(fish::Fish),
        Box::new(kakoune::Kakoune),
        Box::new(yourapp::YourApp),
    ]
}
```

5. Add a `just themes apply-yourapp` recipe if your application supports
   live-reloading.

### The test pattern

Each generator has integration tests in `crates/garden-themes/tests/generators.rs`
that verify:

- The output contains the header comment with palette name and icon
- Color values from the active palette appear in the output
- All expected slots/variables/faces are present
- The relative path is correct

Tests load palettes from the real `_config/palettes.toml` via
`include_str!()`, so they always test against the actual built-in palettes.
